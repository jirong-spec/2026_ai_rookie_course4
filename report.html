各 Lab 紀錄

Lab 0 環境確認與第一次生成 完成 ✓

載入預訓練因果語言模型，用 Chat Template 組 prompt，跑一次 generate，確認 GPU、套件、HuggingFace 連線都正常。也是認識 uv 環境管理工具的起點。

💡 重點：uv run 取代 source .venv/bin/activate，依賴統一宣告在 pyproject.toml，環境可重現。
Lab 1 Chat Template 與訊息結構檢查 完成 ✓

把 JSON 格式的對話（role + content）餵進 tokenizer.apply_chat_template，轉成模型實際接收的字串（含 special tokens、角色標記）。並實作一致性檢查，驗證 system 存在、role 順序正確、content 不為空。

💡 最重要的區別：
訓練用 add_generation_prompt=False（完整對話，含 assistant 回覆）
推理用 add_generation_prompt=True（讓模型接著生成）
搞錯這個，模型表現會大幅下降 — 這就是 Lab6 消融實驗要驗證的事。
Lab 2 Tokenizer 比較與訓練預算估算 完成 ✓

對同一段文字，用多個模型的 Tokenizer 分別計算 raw token 數與套 Chat Template 後的 token 數。並實作訓練預算估算器：

total_tokens = 筆數 × (avg_prompt_tokens + avg_resp_tokens) × epochs
train_hours  = total_tokens / tokens_per_sec / 3600
💡 同樣的中文，不同模型的 token 數可以差 2~3 倍。訓練前先估算成本，是工程實務的基本習慣。

💡 消融實驗的用途：隔離單一變數，量化它的影響。這是 ML 工程中很重要的思維——不要靠直覺，要靠數字說話。

核心觀念整理

🔑 Chat Template 一致性

訓練和推理必須用完全相同的 template。add_generation_prompt 的 True/False 差一個字，模型行為天差地遠。

🔑 LoRA / QLoRA

不改原始權重，插入低秩矩陣。Adapter 只有幾 MB，4-bit 量化大幅降低顯存需求，是現在 fine-tuning 的標準做法。

🔑 SFT 資料格式

每筆資料是 messages 列表（role + content），訓練時完整對話都要餵進去，讓模型學會「給什麼問題，回什麼格式」。

🔑 Offline 評估

用規則或 LLM-as-judge 自動打分，不需人工標記就能快速迭代。本次用四條規則，未來可升級為 BLEU / ROUGE。

🔑 消融實驗

每次只改一個變數，用數字量化它的影響。是驗證工程決策是否有效的最直接方式。

🔑 Token 成本意識

同樣的文字，不同 tokenizer 切出的 token 數差很多。訓練前先估算 token 數和時間，是工程實務的基本功。
